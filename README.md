# Billing Scheduler

Шедулер биллинговых задач — бэкенд на FastAPI + PostgreSQL, фронтенд на Next.js.

## Как запустить

### Что нужно установить заранее
- Python 3.11+
- Node.js 18+
- PostgreSQL 15+

### 1. База данных

Установить PostgreSQL, открой pgAdmin и выполнить:

```sql
CREATE USER billing WITH PASSWORD 'billing';
CREATE DATABASE billing OWNER billing;
```

### 2. Бэкенд

```bash
cd backend
python -m venv venv


venv\Scripts\activate


pip install -r requirements.txt
python -m uvicorn app.main:app --reload --port 8000
```

### 3. Воркер (второй терминал)

```bash
cd backend
venv\Scripts\activate
python -m app.worker
```

### 4. Фронтенд (третий терминал)

```bash
cd frontend
npm install
npm run dev
```

После запуска открыть:
- http://localhost:3000 — интерфейс
- http://localhost:8000/docs — Swagger с документацией API

## Что реализовано

### Бэкенд
- Три таблицы: `clients`, `scheduled_jobs`, `job_runs`
- Воркер опрашивает БД раз в минуту и выполняет задачи по расписанию
- При создании клиента автоматически планируются задачи на текущий и следующий месяц
- Учёт часовых поясов: `billing_day` конвертируется из локального времени клиента в UTC

### Три типа задач

`GENERATE_INVOICE` Генерирует счёт  В день биллинга клиента 
`SEND_REMINDER` Отправляет напоминание (mock POST)  +7 дней после счёта 
`BLOCK_SERVICE` Переводит клиента в статус PAUSED  +30 дней после счёта 

### Фронтенд
- Страница «План на 7 дней» — кнопки ручного запуска и отмены задачи
- Страница «История за 30 дней» — фильтры по статусу и типу, кнопка перезапуска упавшей задачи

### REST API
```
GET  /api/clients/              — список клиентов
POST /api/clients/              — создать клиента
GET  /api/jobs/upcoming         — план на 7 дней
GET  /api/jobs/history          — история за 30 дней (фильтры: status, job_type)
POST /api/jobs/{id}/run-now     — ручной запуск задачи
POST /api/jobs/{id}/cancel      — отмена задачи
GET  /health                    — проверка работоспособности
```

## Допущения

- `billing_day` принимает значения от 1 до 28 — февраль и короткие месяцы не обрабатываются отдельно
- `GENERATE_INVOICE` только логирует факт создания счёта, реальный PDF не генерируется
- `SEND_REMINDER` и алерты отправляются mock POST-запросом на httpbin.org
- Воркер polling-based (раз в минуту), а не event-driven — для биллинга этого достаточно
- Тестовые клиенты создаются автоматически при первом запуске воркера

## Как гарантируется идемпотентность

Идемпотентность реализована на уровне базы данных через уникальный составной индекс:

```sql
UNIQUE (client_id, job_type, target_date)
```

Это означает что одна и та же задача — для одного клиента, одного типа и одной даты — физически не может существовать в БД дважды. Если воркер упал посреди работы и перезапустился, повторный `INSERT` упадёт с `IntegrityError`, транзакция откатится, и дубля не будет.

Это `at-least-once` семантика с защитой от дублирования через constraint на уровне БД, а не в логике приложения — потому что логику можно обойти при race condition, а индекс — нет.

---

## Retry-логика с экспоненциальным backoff

Если задача упала — воркер не бросает её, а повторяет с нарастающей задержкой:

```
Попытка 1 упала → retry через 1 минуту
Попытка 2 упала → retry через 5 минут
Попытка 3 упала → статус FAILED + алерт (mock POST на httpbin.org)
```

Количество попыток хранится в поле `retry_count`, время следующего запуска — в `next_run_at`. Каждая попытка записывается в таблицу `job_runs` с результатом и сообщением об ошибке.

## Trade-offs

**PostgreSQL как очередь задач вместо Redis/RabbitMQ**

Я выбрал хранить задачи прямо в PostgreSQL, а не поднимать отдельный брокер. Плюсы: меньше инфраструктуры, ACID-гарантии, идемпотентность через уникальный индекс. Минус: при очень большом количестве задач (десятки тысяч в минуту) это станет узким местом. Для биллинга с сотнями клиентов — нормальное решение.

**Polling раз в минуту вместо event-driven**

Воркер сам спрашивает БД "есть ли что запустить?" каждые 60 секунд. Это проще чем подписываться на события, и для биллинга точность до минуты — достаточная.

**`create_all()` вместо Alembic миграций**

Для простоты запуска таблицы создаются автоматически при старте. В продакшне это неприемлемо — нужен Alembic для контролируемых миграций.

## Что не успел и почему

- **Alembic миграции** — использовал `create_all()` чтобы упростить первый запуск
- **Тесты** — не хватило времени, особенно нужны тесты на `scheduler.py` с edge cases по датам
- **Авторизация** — по заданию не требовалась
- **Реальная генерация PDF счетов** — замокано логированием
